4) UNIVERSE BUILDING LOGIC (06:15 AM JOB)
-----------------------------------------
Job: universe-refresh-append-backfill

What it does:
1. Fetches Upstox raw universe snapshot (BOD instrument master)
2. Filters to eligible tradeable rows (NSE cash universe policy)
3. Appends ONLY new eligible symbols to "🧾 Universe Instruments"
4. Starts one backfill pass for newly appended symbols (priority first)

Why append-only:
- Stable universe does not get rebuilt every day.
- Only new symbols are added.
- Existing scores/history are preserved.

5) HISTORY / CANDLE DATA LOGIC
------------------------------
There are two major modes:

A. One-time full history backfill (bootstrap / on-demand)
- Purpose: build enough history for scoring (policy threshold = 320 bars)
- Not scheduled continuously anymore
- Used when universe is first built or when many new instruments appear

B. Daily latest candle update (morning retries after Upstox publishes official 1D)
- Job: score-cache-update-close
- Runs at 07:05 / 07:25 / 07:45 / 08:05 IST
- Fetches OFFICIAL daily 1D candle from Upstox (no provisional intraday storage)

Why multiple retries?
- Upstox daily 1D candle may not be available for all symbols exactly at 7:00.
- Morning retries improve freshness before scoring.

Current decision (your choice):
- No provisional/intraday daily candle generation.
- Use only official Upstox daily candle.


5.1) HISTORY STATUS MEANINGS (History Candle 1D)
-----------------------------------------------
Common statuses:
- FRESH_READY
  Latest expected daily candle is present and row is score-ready (>= min bars)

- INSUFFICIENT_HISTORY_FINAL
  The symbol has official candles, but total bars are still below threshold
  (Need=320). It is terminal for now (not a bug).

- STALE_SKIPPED
  Latest expected daily candle not available after retry attempts.
  For the morning retry window, same-day stale rows are now retried again
  (patched behavior), so they are not permanently skipped for the day.

What "prefillComplete=true" means in close-update summary:
- All rows are accounted for as one of:
  - fresh ready, or
  - terminal insufficient history (valid skip), or
  - terminal stale skipped (if provider delay persists)

This does NOT mean every row is fresh-scoreable. It means the batch has
finished processing the full universe for that run.


5.2) IMPORTANT HISTORY METRICS (Job JSON)
----------------------------------------
Returned by score-cache-update-close and related cache prefetch jobs:

- scanned
  How many universe rows were checked

- fetches
  How many provider fetch operations were performed

- updated
  How many cache files changed (bars or last candle changed)

- retriedNoChange
  Fetch happened but cache did not change (provider still same data)

- freshReady
  Rows with latest candle present and score-ready

- terminalInsufficientHistory
  Rows with official data but < min bars (valid skip)

- terminalStaleSkipped
  Rows still stale after retries (provider delay / unavailable latest candle)

- prefillComplete
  True when all rows are classified for the run (fresh/terminal)

- staleOrMissing
  Remaining unprocessed/pending rows for the current run


6) SCORING LOGIC (08:30 AM JOB)
-------------------------------
Job: score-refresh

Current mode (finalized):
- cache_only = true
- require_fresh_cache = true
- fresh_hours = 0 (force scoring run, do not skip by recent-score window)

Meaning:
- Score is computed from the cached daily candles only.
- If a row's latest daily candle is stale (not current), it is skipped.
- If a row has insufficient history (<320 bars), it is skipped.
- No provider candle fetch is attempted in this scoring step.

Important:
- Score job still fetches MARKET REGIME (Nifty/VIX/PCR/FII-DII) first.
- If Upstox token is expired, score-refresh fails unless token is renewed.


6.1) SCORE PREREQUISITES (PER STOCK)
------------------------------------
A stock is scored only if:
1. Enabled = Y
2. It has enough daily history bars (policy min_bars = 320)
3. Latest daily candle is current (when require_fresh_cache=true)
4. Indicators can be computed

Indicator engine technical minimum:
- compute_indicators() needs ~80 bars
But current scoring policy still requires:
- 320 bars minimum (quality policy)

This is why some stocks are skipped even though some indicators could exist.


6.2) WHAT THE UNIVERSE SCORE MEANS
----------------------------------
Universe score is a technical attractiveness score (0 to 100) for ranking
stocks for watchlist consideration.

It is NOT:
- valuation score
- fundamentals score
- news/earnings score
- intraday execution score

Think of it as:
"How strong is this stock technically right now based on daily structure,
 momentum, breakout context, and volume confirmation?"


6.3) SCORE FORMULA (UNIVERSE SCORE BREAKDOWN)
---------------------------------------------
The score is built from multiple parts and then clamped to 0..100.

Positive parts:
- E = EMA trend structure
- P = Price position vs moving averages
- R = RSI quality zone
- M = MACD strength / buy cross
- B = Breakout / 52-week high context
- V = Volume ratio confirmation
- O = OBV (On-Balance Volume) trend confirmation
- U = User priority bonus (manual PM override from Universe sheet)

Negative part:
- N = Penalties (overbought / weak candle / far from highs, etc.)

Final:
- S = final score after adding everything and applying clamp

Formula (conceptual):
S = clamp( E + P + R + M + B + V + O + N + U, 0, 100 )


6.4) SC CALC (COLUMN Z) - ABBREVIATIONS
---------------------------------------
Example:
E20|P10|R15|M5|B25|V15|O5|N0|U0|S95

Meaning:
- E20 : strong EMA trend alignment (fast > medium > slow)
- P10 : price above EMA20 and EMA50
- R15 : RSI in preferred bullish zone
- M5  : MACD supportive (positive histogram)
- B25 : breakout/near-high strength very strong
- V15 : high volume confirmation
- O5  : OBV rising
- N0  : no penalties
- U0  : no manual priority bonus
- S95 : final score 95

Skip cases in Sc Calc use compact codes:
- IH:171/320   = insufficient history (171 bars available, need 320)
- STL:YYYY-MM-DD = stale cache skipped
- IND0:YYYY-MM-DD = indicators unavailable


6.5) LAST NOTE (COLUMN R) - ABBREVIATIONS
-----------------------------------------
Example:
Score=95|Reg=TREND|Bias=NEUTRAL|RSI=62.4|VR=7.83|LCD=2026-02-25|
Src=gcs_score_cache_1d_partial_history|Prov=N

Meaning:
- Score = final universe score
- Reg   = market regime at scoring time (TREND / RANGE / AVOID)
- Bias  = market bias from Nifty move (BULLISH / BEARISH / NEUTRAL)
- RSI   = latest RSI used in scoring
- VR    = volume ratio (current volume / avg volume)
- LCD   = Last Candle Date used for scoring
- Src   = source of cached candle data used
- Prov  = provisional candle used? (Y/N)
        In your current flow this should be N (official daily only)


6.6) HOW EACH SCORE COMPONENT IS CALCULATED (PM SIMPLE VERSION)
---------------------------------------------------------------
E = EMA Trend Structure (Trend health)
- +20 if strong bullish EMA stack (fast > med > slow)
- +10 if partial bullish structure (EMA20 > EMA50)

P = Price Position (Where price sits)
- +5 if price > EMA20
- +5 if price > EMA50

R = RSI (Momentum quality)
- +15 if RSI in preferred strength zone (50-65)
- +8  if RSI is improving but weaker (40-50)
- +5  if RSI is high but still acceptable (65-75)

M = MACD (Momentum confirmation)
- +5 if MACD histogram > 0
- +5 if MACD fresh BUY cross

B = Breakout / 52-week high context
- +15 if breakout
- +10 if near breakout
- +8 if within 10% of 52-week high (and no breakout)

V = Volume confirmation
- +15 if volume ratio >= 1.5x
- +10 if volume ratio >= 1.2x
- +5 if volume ratio >= 1.0x

O = OBV confirmation
- +5 if OBV is rising

N = Penalties (Risk/warning signs)
- -15 if RSI > 80 (overbought)
- -15 if RSI < 35 (very weak)
- -5  doji candle
- -5  bearish candle
- -10 far (>30%) from 52-week high

U = User priority bonus (manual PM override)
- +0 to +5 from Universe column "Priority"

6.8) WHY MANY SCORES ARE ZERO (NOT A BUG)
-----------------------------------------
A score of 0 is often a valid result.

Example pattern:
- weak trend (E/P low)
- poor breakout context (B low)
- some penalties (N negative)
- final score falls below 0 and gets clamped to 0

This is healthy behavior for a ranking system:
- It strongly rejects weak setups
- It helps keep the watchlist focused

8) WATCHLIST CREATION LOGIC (09:16 AM JOB)
------------------------------------------
Job: watchlist-refresh

Current final behavior:
- Uses live market regime (Nifty/VIX/PCR/FII-DII source chain)
- Filters only rows with valid scores > 0
- Excludes rows where Last Note starts with "Skip="
- Requires rows to be scored TODAY (require_today_scored=true)
- Does NOT require 100% score coverage (require_full_coverage=false)

Why this is good:
- You do not get blocked by a few delayed/insufficient symbols
- You still ensure the watchlist uses today's scoring run

Selection process (simple):
1. Build eligible pool from scored rows
2. Filter by min score (current min_watchlist_score=1)
3. Apply product resolution (CNC/MIS)
4. Diversify selected list (MIS/CNC mix)
5. Write top N (current target = 300) to "📋 Watchlist"

9) OPERATIONAL DEPENDENCY: UPSTOX TOKEN (IMPORTANT)
---------------------------------------------------
Current Upstox auth flow is semi-automated:
1. Job requests token at 03:35 AM
2. User manually approves via Upstox/notifier flow
3. Upstox calls webhook /webhooks/upstox/access-token
4. Token is stored automatically in Secret Manager

Important limitation:
- Daily manual approval is typically still required with this official API flow
  (no refresh-token auto-renew in the current retail flow used here).

Impact:
- If token is not approved, score-refresh/watchlist-refresh can fail because
  market regime fetch happens before scoring/watchlist generation.

Current mitigation:
- Morning retries for candles
- Easy manual token-request endpoint

Potential future hardening (recommended):
- score-refresh: neutral-regime fallback when token unavailable
- keep watchlist-refresh strict for live regime

12) HOW TO READ SCORES AS A PRODUCT MANAGER
-------------------------------------------
Simple score interpretation:

80-100 : Very strong technical setup
60-79  : Good setup (high watchlist priority)
40-59  : Mixed but possible (secondary candidates)
1-39   : Weak / early / low conviction
0      : Reject for now (or heavily penalized)
Skip=* : Not scoreable under current policy (usually insufficient history)

Rule of thumb:
- High score = technical quality + confirmation + no major penalties
- Low/zero score = weak structure, weak momentum, or risk flags

  Full set of eligible stocks the system can consider

17) FINAL PM SUMMARY (ONE SCREEN)
---------------------------------
Every morning the system:
1. Refreshes the trade universe and appends any new stocks
2. Starts backfill only for new stocks
3. Tries multiple times to fetch the latest official daily candle from Upstox
4. Scores all stocks with fresh daily data (skips insufficient-history/stale rows)
5. Builds a 09:16 watchlist using live regime data, today's scores only,
   without waiting for 100% universe coverage
